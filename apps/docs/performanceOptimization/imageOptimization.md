# 现代图片资源调度与体验优化

**核心本质**：图片优化是一场围绕“**像素是否绝对必要、何时进入下载队列、以何种体积传输、上屏后是否引起布局坍塌**”的系统级资源调度。在现代 Web 应用中，图片往往占据了页面体积的绝对大头，更是拖垮 LCP (首屏绘制) 和引爆 CLS (布局偏移) 的头号“**战犯**”。

**战术纪律**：LCP 英雄图必须最高优、最精确、最稳定；非首屏图片必须延后、被裁剪、被按需加载。永远不要让移动端去下载一张 PC 端的 4K 原图。

## 1. 图片性能的四个瓶颈

当一张图片加载“**慢**”时，往往不是单一的网络问题，而是以下环节的致命叠加：

* **编解码落后 (Format)**：在摄影大片上依然顽固使用未压缩的 JPEG，或用巨型 PNG 承载不需要透明通道的背景图。
* **物理像素越级 (Dimension)**：在 CSS 尺寸仅为 `300x200` 的容器里，强塞了一张真实分辨率为 `3000x2000` 的图片，不仅白白浪费数十倍网络带宽，更在浏览器绘制时疯狂榨干 CPU 和内存（解码耗时）。
* **发现时机滞后 (Discovery)**：LCP 关键图片被“**藏**”在了 CSS 的 `background-image` 中，或者由深埋在 JS 瀑布流底部的组件去动态创建 DOM。这会导致浏览器迟迟不知道该请求什么。
* **空间占位缺席 (Space Layout)**：在 HTML 中没有写明尺寸，导致图片在下载完成后瞬间“**撑开**”页面，制造毁灭性的 CLS 体验。

## 2. 格式选择与编码策略

格式的选择不再是“**唯压缩率论**”，而是要在网络传输体积、客户端设备解码成本以及浏览器兼容性之间寻找最优解。

* **AVIF (新王登基)**：基于 AV1 视频编码，在低码率下保留惊人细节，极其适合首屏大图。但极低端设备解码较耗电。
* **WebP (行业中坚)**：兼容性极佳的“**万金油**”，压缩率和画质远超 JPEG，且支持透明通道，是绝大多数业务图的首选。
* **PNG (精准打击)**：仅在绝对需要完美透明边界的 UI 切图、Logo 或代码截图等纯色块场景使用。
* **SVG (矢量双刃剑)**：适合单色 Icon 和基础插画。但如果将超高复杂度的设计稿强行导出为 SVG，其成千上万个 `<path>` 节点会带来灾难级的 DOM 解析与主线程阻塞成本。

**最佳实践：利用 `<picture>` 实现优雅降级**

```html
<picture>
  <source srcset="/hero.avif" type="image/avif" />
  <source srcset="/hero.webp" type="image/webp" />
  <img src="/hero.jpg" width="1200" height="640" alt="活动封面" />
</picture>

```

## 3. 响应式图片与尺寸治理

不要企图用一张“**万能高清大图**”去适配所有设备。你必须结合图片 CDN（如阿里云 OSS、七牛云的图片处理参数）进行动态下发。

**利用 `srcset` + `sizes` 将选择权交还给浏览器：**

```html
<img
  src="/product-800.jpg"
  srcset="/product-400.jpg 400w, /product-800.jpg 800w, /product-1600.jpg 1600w"
  sizes="(max-width: 768px) 100vw, 50vw"
  width="800"
  height="600"
  alt="响应式商品图"
/>

```
* **按布局出图**：图片 CDN 应支持宽度、质量、格式参数。
* **按 DPR 选择**：高分屏需要更清晰，但不能无脑返回超大图。
* **限制上传原图直出**：用户上传图片必须经过压缩、裁剪、转码流水线。
* **服务端生成多规格**：常用卡片、详情、缩略图尺寸应固定规格。

**工作机制**：`sizes` 提前告诉浏览器：“在手机端这张图会占满屏幕 (100vw)，在电脑端它占半屏 (50vw)”。浏览器会结合当前屏幕的物理宽度和 DPR (设备像素比)，在 `srcset` 中精准挑出最省带宽的那张图去下载。

## 4. LCP 图片优先级

对于决定生死的那张 LCP 首图，常规的加载方式是不够的，必须“**特事特办**”。

* **严禁滥用懒加载**：绝对不要在首屏可见图片上加上 `loading="lazy"`，它会强行推迟关键资源的下载顺位。
* **高优特权 (Fetch Priority)**：直接在 `<img>` 标签上注入 `fetchpriority="high"`，命令浏览器立刻开辟通道。
* **背景图的救赎**：如果 LCP 必须是 CSS `background-image`，请务必在 HTML `<head>` 中用 `preload` 提前宣誓主权，否则网络请求将被滞后到 CSSOM 构建完毕之后。

```html
<link rel="preload" as="image" href="/hero.avif" fetchpriority="high" />
```

## 5. 极致懒加载与内存释放

非首屏图片必须为关键资源让路，但在实现按需加载时要有极客的讲究。

* **原生降维打击**：利用原生属性 `loading="lazy"` 处理长列表、底部评论区的零碎图片，把复杂性交给浏览器。
* **精准拦截 (IntersectionObserver)**：针对极致的瀑布流或定制化的懒加载（需考虑提前 1-2 屏预加载），摒弃极其消耗性能的 `window.onscroll` 事件监听。

:::details  
```js
// 1. 创建一个交叉观察器实例 (IntersectionObserver)
// 不要一开始就加载所有图片，而是等图片即将进入用户视野时，再将真实的图片地址赋值给它，从而节省大量首屏网络带宽。
const observer = new IntersectionObserver((entries) => {
    // entries 是一个数组，包含了当前正在被观察，且“交叉状态”发生变化的元素集合
    // 即使只观察一个元素，它也是个数组
    entries.forEach((entry) => {
        // 2. 核心判断：如果该元素没有进入可视区域 (isIntersecting 为 false)，则直接退出，什么都不做。
        // 这说明它可能还在屏幕下方，或者用户刚刚把它滑出屏幕。
        if (!entry.isIntersecting) return;
        // 3. 执行懒加载逻辑：走到这里，说明该元素已经进入了可视区域
        const img = entry.target; // 获取触发事件的具体 DOM 节点（这里就是那个 <img> 标签）
        // 将真实图片的 URL（存放在 data-src 属性中）赋值给 src 属性
        // 此时，浏览器才会真正发起网络请求去下载这张图片
        img.src = img.dataset.src;
        // 4. 性能优化关键：停止观察 (unobserve)
        // 既然这张图片已经加载过了，就不需要再继续监视它了。
        // 如果不解除观察，用户上下滑动时依然会不断触发回调，浪费 CPU 性能。
        observer.unobserve(img);
    });
});
// 5. 初始化绑定
// 选中页面上所有带有 data-src 属性的图片（这些图片通常一开始 src 为空，或者是一张极小的占位图）
document.querySelectorAll('img[data-src]').forEach((img) => {
    // 把它们一个一个地交给上面创建的 observer 去监视
    observer.observe(img);
});
```
:::

* **图片内存回收 (极客操作)**：在极端的超长商品列表中，不仅要处理进视野的懒加载，对于已经滑出视野几十屏的高清大图，应当将其 `src` 替换为空白图以释放内存，防止客户端崩溃 (OOM)。

## 6. CLS 防御矩阵与解码隔离

上屏必须稳如磐石，绝不能让阅读中的用户遭遇排版跳动。

* **骨架闭环**：`width` 和 `height` 是最古老也是最有效的防抖武器。配合 CSS 的 `aspect-ratio`，为图片提前搭建“**承重墙**”。

```css
/* 终极防抖占位 CSS 范式 */
.responsive-cover {
  width: 100%;
  aspect-ratio: 16 / 9;      /* 提前撑开高度，杜绝 CLS */
  object-fit: cover;         /* 确保比例稳定，不变形 */
  background-color: #f2f4f7; /* 图片未加载前的骨架底色 */
}
```

* **异步解码降权**：针对处于屏幕下方边缘的非首屏大图，使用 `decoding="async"`。它能确保这张图在后台线程默默解码，绝不抢占主线程执行 JS 和响应点击交互的时间。

```html
<img
    src="/heavy-gallery-image.avif"
    loading="lazy"
    decoding="async"
    width="1200"
    height="800"
    alt="高清画廊图"
/>
```

## 7. 核心监控与审计指标

| 核心指标 | 观测目标与深度含义 | 优化水位线 |
| --- | --- | --- |
| **LCP (最大内容绘制)** | 最能代表这篇文档/这个商品详情的图片，何时完整呈现在用户眼前。 | < 2.5s |
| **Wasteful Bytes (浪费字节)** | 浏览器通过 Network 下载的真实体积，与其最终渲染在屏幕上的物理尺寸之间的溢出比。 | 接近 1:1 (严禁大图小显) |
| **Decoded Size** | 解码后像素是否远超展示尺寸。 | 接近实际渲染尺寸 |
| **CLS (累积布局偏移)** | 图片从“**未下载**”到“**渲染完毕**”的瞬间，是否重创了页面已有的排版。 | < 0.1 |
| **Cache Hit Ratio (缓存命中率)** | 经过动态参数处理后的图片 CDN，在边缘节点的命中情况。 | > 90% |
