# 虚拟列表与大数据渲染

**核心本质**：虚拟列表不是“滚动组件技巧”，而是用极少量 DOM 承载海量数据的渲染降维手段。它通过模拟完整滚动高度，只渲染视口附近的节点，把浏览器从成千上万 DOM 的布局、绘制和 diff 压力中解放出来。

**战术纪律**：列表数据可以很多，真实 DOM 不能很多；滚动必须稳定，测量必须可控，交互状态不能绑死在会被回收的节点上。

## 1. 长列表的性能灾难链路

直接渲染几千到几万条数据，会同时击穿多个环节：

* **DOM 数量过多**：样式计算、布局和绘制成本线性上升。
* **首次渲染慢**：首屏要创建大量真实节点。
* **滚动掉帧**：每一帧都可能触发大量绘制和合成。
* **框架 diff 变重**：状态更新时需要比较和更新大量 VNode。
* **内存和 GC 压力**：节点、事件、闭包和组件实例长期占用内存。

## 2. 固定高度虚拟列表

固定高度是最简单、性能最稳定的虚拟滚动场景。核心思路是根据 `scrollTop` 计算可见区间，再用偏移量把真实 DOM 放到正确位置。

```javascript
const itemHeight = 40;
const visibleCount = Math.ceil(containerHeight / itemHeight);
const start = Math.floor(scrollTop / itemHeight);
const end = start + visibleCount + overscan;
const offsetY = start * itemHeight;
const visibleItems = list.slice(start, end);
```

典型结构如下：

```html
<div class="viewport">
  <div class="spacer">
    <div class="content"></div>
  </div>
</div>
```

```css
.viewport {
  height: 480px;
  overflow: auto;
}

.spacer {
  height: var(--total-height);
  position: relative;
}

.content {
  transform: translateY(var(--offset-y));
}
```

## 3. 动态高度虚拟列表

动态高度列表更接近真实业务，但复杂度也更高。消息流、搜索结果、评论列表常常属于这种情况。

* **预估高度**：先用默认高度渲染，保证滚动条可用。
* **渲染后测量**：元素真实渲染后记录高度。
* **维护前缀和**：用累计高度快速计算某个位置对应的数据项。
* **二分查找起点**：根据 `scrollTop` 在高度表中找到起始项。
* **处理高度变化**：图片加载、折叠展开、字体变化后必须更新缓存。

动态高度最大的风险是滚动跳动。高度修正时要尽量保持用户当前视口中的锚点元素稳定。

## 4. overscan 与滚动体验

`overscan` 表示视口之外额外渲染的项目数量，用于防止快速滚动时露白。

* **太小**：滚动速度快时容易出现空白。
* **太大**：真实 DOM 增多，虚拟化收益下降。
* **动态调节**：可根据滚动速度、设备性能、项高度调整。
* **优先稳定**：业务表格通常宁可多渲染几行，也不要频繁露白。

## 5. 交互与状态陷阱

虚拟列表会复用和回收 DOM，因此交互状态必须从 DOM 生命周期中解耦。

* **稳定 key**：item 必须使用业务唯一 ID，不要使用数组下标。
* **弹层外置**：下拉菜单、tooltip、弹窗不要绑死在会被回收的 item DOM 上。
* **焦点管理**：键盘导航和输入框编辑要处理节点回收后的焦点恢复。
* **选中态外置**：选中、展开、编辑状态应放在独立状态表中。
* **可访问性**：超大列表仍要考虑 `aria-rowcount`、焦点顺序和屏幕阅读器体验。

## 6. 何时不要虚拟化

虚拟化也有成本，不是所有列表都应该引入。

* **数据量很小**：几十条数据直接渲染更简单。
* **SEO 强依赖内容**：虚拟列表可能让爬虫看不到完整内容。
* **全文搜索依赖 DOM**：浏览器原生页内搜索无法搜索未渲染项。
* **高度极端不稳定**：复杂动态布局会让测量和锚点维护成本过高。

## 7. 核心度量指标

| 核心指标 | 观测目标与含义 | 优化达标线 |
| --- | --- | --- |
| **DOM Nodes** | 页面真实 DOM 数量是否受控。 | 长列表稳定在数百级 |
| **FPS** | 滚动时是否掉帧。 | 接近 60fps |
| **Long Task** | 滚动和更新是否产生长任务。 | 尽量为 0 |
| **Memory** | 列表滚动后内存是否持续上涨。 | 无明显泄漏 |
| **Blank Rate** | 快速滚动时是否露白。 | 业务可接受范围内 |
