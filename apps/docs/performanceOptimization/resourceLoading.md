# 资源加载优化

**核心本质**：资源加载优化的终极奥义，并非将所有静态资源一股脑地提前塞给浏览器，而是充当一名精密的“**交通指挥官**”。你需要指挥浏览器在极为有限的带宽、有限的并发连接数以及单线程的主线程时间里，优先服务那些真正决定首屏存亡和核心交互的资源。关键资源晚一秒，LCP（最大内容绘制）就晚一秒；而非关键资源如果抢跑，关键渲染路径（Critical Rendering Path）就会被无情挤占，导致页面“**假死**”。

**战术纪律**：当前页面立刻要用的资源，使用 `preload` 抢占；未来极大概率要用的资源，使用 `prefetch` 捡漏；第三方跨域的关键域名，使用 `preconnect` 铺路。精准克制，绝不滥用。

## 1. 浏览器资源调度链路

现代浏览器极其聪明，但它再聪明也需要时间去解析和推断。理解浏览器的内部调度链路，是你介入优先级控制的前提：

* **资源发现时机**：浏览器的 HTML 流式解析器、CSS 词法分析器以及 JavaScript 引擎的执行，都会在不同阶段暴露出新的资源请求。
* **网络建连损耗**：每一次全新的跨域请求，都不可避免地经历 DNS 查询、TCP 三次握手和极其耗时的 TLS 密文协商。
* **启发式排队与降级**：浏览器内部有一套默认的优先级评分机制（Highest, High, Medium, Low, Lowest）。通常，HTML 文档、关键 CSS 和视口内的图片会被赋予极高优先级；而异步脚本、视口外的图片则会被自动降级。
* **下载解码与执行**：资源下载完毕仅仅是开始。图片的解码、字体的光栅化、JavaScript 的 V8 编译，都会继续疯狂消耗 CPU 资源。
* **木桶效应**：CSS（构建 CSSOM）、定制字体（FOUT/FOIT）、LCP 英雄图片以及主入口 JS，这四者是决定首屏生死的核心要素。

## 2. 抢占当前页面关键资源

`preload` 是开发者向浏览器下达的强制指令：“**不要等解析到对应的 DOM 才去下载，现在、立刻、马上把这个资源拉下来，我保证页面很快就会用到它。**”

```html
<link rel="preload" href="/fonts/main.woff2" as="font" type="font/woff2" crossorigin="anonymous" />

<img src="/img/hero.avif" fetchpriority="high" alt="Hero Banner" />
```

* **适用靶点**：首屏最大内容绘制（LCP）图片、防闪烁的关键字体、当前路由极其关键的 CSS 文件，以及那些深藏在 CSS 文件内部、浏览器极难提前发现的背景大图。
* **类型声明（as 属性）**：必须绝对准确。错误的类型声明会导致浏览器使用不同的请求上下文（Request Context），最终导致资源被重复下载两次，严重浪费带宽。
* **跨域属性（crossorigin）**：字体文件即使在同源策略下，规范也要求必须以匿名跨域模式请求。漏写 `crossorigin` 是导致字体被重复拉取的头号元凶。
* **克制与收敛**：如果把所有资源都打上 `preload`，就等于没有优先级。滥用会导致浏览器并发数爆满，反而把真正的关键资源堵在网络队列里。

## 3. 预取未来高概率资源

如果说 `preload` 是“**雪中送炭**”，那么 `prefetch` 就是“**锦上添花**”。它的优先级极低，专门利用浏览器渲染完毕后的网络空闲期，静默下载未来可能会用到的资源。

```html
<link rel="prefetch" href="/assets/dashboard-chunk.js" as="script" />

<script type="speculationrules">
{
  "prerender": [
    { "source": "list", "urls": ["/next-page"] }
  ]
}
</script>
```

* **黄金场景**：多步表单的下一步逻辑、登录成功后必然跳转的工作台页面、鼠标悬停（Hover）在导航菜单上时的目标路由包裹。
* **严格隔离首屏**：绝不允许用 `prefetch` 去拉取当前首屏的关键资源，这会导致首屏加载出现毁灭性的延迟。
* **弱网环境的克制**：在 3G 或极差的移动网络下，低命中率的预取会疯狂消耗用户的有限流量和手机电量。成熟的框架（如 Next.js/Nuxt）通常会结合 `navigator.connection` API，在 4G/WIFI 环境下才开启预取。
* **数据驱动预判**：不要靠产品经理的直觉去预取。应当基于真实用户的数据埋点（如马尔可夫链模型），精准算出点击转化率最高的下一个页面。

## 4. 缩短第三方建连耗时

第三方域名（如 CDN、统计、客服系统）的第一个请求，往往要承受完整的 DNS + TCP + TLS 的“**三连暴击**”，耗时动辄几百毫秒。提前打通网络管道，是优化第三方加载的最佳手段。

```html
<link rel="preconnect" href="https://cdn.example.com" crossorigin />

<link rel="dns-prefetch" href="//static.example.com" />

```

* **preconnect 策略**：极其适合那些承载了首屏核心图片、核心字体或是核心 API 接口的跨域服务器。它会在后台静默完成漫长的 TLS 握手。
* **dns-prefetch 策略**：由于不建立 TCP 链接，几乎没有成本。可以把页面内所有潜在的第三方域名全部列上。
* **严格控制数量**：浏览器同时维持的 TCP 闲置连接数是有上限的（通常单域名 6 个）。如果给十几个域名建立 `preconnect` 却迟迟不发送真实请求，连接会在 10 秒左右被浏览器主动切断，你的优化心血将付之东流。

## 5. Script 加载策略

JavaScript 的下载与执行时机，直接决定了 HTML 解析器是否会被强制挂起（Blocking）。

```html
<script src="/legacy-jquery.js"></script>

<script src="/analytics.js" async></script>

<script src="/main.js" defer></script>

<script type="module" src="/src/main.js"></script>
```

* **async（异步执行）**：谁先下载完谁就立刻抢占主线程执行。它完全破坏了脚本之间的依赖顺序，极其适合那些与业务逻辑完全解耦的独立脚本（如 Google Analytics、各种打点探针）。
* **defer（延迟执行）**：它向浏览器保证：“我只在后台默默下载，绝对不打断 HTML 解析。等 DOM 树彻底构建完毕（`DOMContentLoaded` 触发前），我再严格按照 `<script>` 标签书写的先后顺序依次执行。” 这是目前前端入口脚本的标准答案。

## 6. 字体、第三方资源与主线程解放

第三方资源（客服、广告、埋点、地图）往往是首屏性能崩溃的罪魁祸首，它们常常肆无忌惮地抢占关键渲染路径。

* **字体加载体验（FOIT vs FOUT）**：使用 `font-display: swap`，告诉浏览器如果自定义字体还没下载完，先用系统默认字体兜底显示文字，绝不让用户盯着一块空白屏幕发呆。
* **第三方脚本的终极延后**：那些不影响用户核心体验的探针或客服代码，必须延后到页面完全空闲时加载。可以利用 `window.onload` 结合 `requestIdleCallback` 实现极致的错峰执行。

:::details
```js
window.addEventListener('load', () => {
  requestIdleCallback(() => {
    import('./analytics.js');
  });
});
```
:::

* **拥抱 Web Worker (Partytown)**：对于重度消耗主线程的第三方脚本，可以使用类似于 Partytown 这样的先进库，将它们全部放逐到 Web Worker 中运行，彻底解放浏览器的渲染主线程。
* **错误沙箱隔离**：第三方资源的超时、崩溃或是接口 500 报错，必须被异常捕获（Try/Catch 或全局 Error Boundary）拦截，绝不能牵连主业务线白屏。

## 7. 核心度量指标与水位线

优化效果必须用冷酷的数据来验收。以下是资源调度优化的核心监控矩阵：

| 核心度量指标 | 观测目标与深度解析 | 极客级优化达标线 |
| --- | --- | --- |
| **LCP Resource Delay** | 最大内容绘制资源的发现是否太晚？如果 HTML 解析了很久才开始请求主图，说明缺乏 `preload`。 | **接近于 0ms** (HTML 收到后立即发起请求) |
| **Request Waterfall** | 关键网络请求瀑布流中，是否存在严重的串行排队、等待或是资源阻塞现象。 | **消除阶梯状长链**，实现首屏核心资源高并发 |
| **TTFB (首字节时间)** | 服务端响应、DNS 解析以及连接建立的耗时是否严重拖垮了后续所有资源的加载起点。 | **< 800ms** (边缘节点交付更应 < 200ms) |
| **Unused Preload** | 你的“**预判**”是否失误？所有被 preload 但在页面加载后几秒内未被使用的资源，都是严重的性能事故。 | **绝对值为 0** (100% 精准命中) |
| **Third-party Blocking** | 第三方不可控资源（广告/监控）占用主线程的时间总和，是否严重影响了 INP 交互延迟。 | **首屏关键路径内为 0ms** |
