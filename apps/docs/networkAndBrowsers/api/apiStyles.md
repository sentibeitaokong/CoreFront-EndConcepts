# REST、GraphQL、RPC：API 架构风格与工程实践

REST、GraphQL、RPC 并非单纯的 URL 拼写规范，而是决定了系统**数据交换模型、前后端协作边界、网络传输效率以及整体微服务治理策略**的核心架构风格。

## 1. 核心维度深度对比

| 架构风格    | 核心抽象                  | 底层与载荷                | 典型入口               | 核心优势                                                                | 致命代价与挑战                                                                         |
| ----------- | ------------------------- | ------------------------- | ---------------------- | ----------------------------------------------------------------------- | -------------------------------------------------------------------------------------- |
| **REST**    | **资源 (Resources)**      | HTTP(s) + JSON/XML        | `/api/v1/users/123`    | 语义极其清晰，完美契合 HTTP 协议，天然支持 CDN 与边缘缓存。             | 强依赖多接口组合，极易产生 Over-fetching（过度获取）或 Under-fetching（获取不足）。    |
| **GraphQL** | **图与字段 (Data Graph)** | HTTP (通常 POST) + JSON   | `/graphql`             | 客户端声明式按需获取，一次请求实现复杂聚合，自带强类型 Schema。         | 颠覆了 HTTP 缓存语义；极易引发数据库 N+1 查询风暴；细粒度权限控制极其复杂。            |
| **RPC**     | **动作与方法 (Actions)**  | TCP/HTTP2 + Protobuf/JSON | `/UserService/GetUser` | 屏蔽网络细节（像调用本地函数一样），强类型约束，序列化极快（如 gRPC）。 | 彻底丧失 HTTP 通用语义（方法、状态码）；与浏览器生态存在天然隔阂（通常需要网关代理）。 |

## 2. REST

REST 的本质是**表现层状态转化**，它强迫开发者将系统能力抽象为静态的“**资源名词**”，并通过标准的 HTTP 动词驱动状态流转。

```http
GET    /users          # 获取用户列表
POST   /users          # 创建用户
GET    /users/123      # 获取特定用户详情
PUT    /users/123      # 幂等替换用户全部状态
PATCH  /users/123      # 局部更新用户状态
DELETE /users/123      # 销毁资源
```

### 2.1 REST 的高维优势

- **缓存极致利用**：GET 请求天然具备幂等性，完美契合浏览器、CDN、Nginx 的缓存机制（通过 `ETag`、`Cache-Control`）。
- **Richardson 成熟度模型**：成熟的 RESTful API（Level 3 / HATEOAS）会在响应中返回 `_links`，由服务端动态下发客户端下一步可执行的操作，实现真正的解耦。
- **无状态性 (Stateless)**：请求之间互相独立，服务端不保存上下文，极大降低了集群水平扩容的难度。

### 2.2 REST 的工程痛点

- **过度获取**：列表页只要姓名，却返回完整用户详情。
- **获取不足**：页面需要用户、订单、优惠券，可能要发多个请求。
- **接口膨胀与胶水层**：为了满足 PC 端、移动端、手表端不同的数据结构需求，常常被迫开发 `/v1/users/detailForApp` 等畸形接口，最终催生了 BFF (Backend For Frontend) 层的必然出现。
- **无脑 CRUD 化**：对于“**重置密码**”、“**审批流转**”等强动作属性的业务，强行抽象为资源（如 `POST /password-resets`）显得僵硬且违背直觉。

## 3. GraphQL

GraphQL 是 Facebook 提出的一种数据查询语言，其核心理念是“**请求你所需要的，不多也不少**”。

```graphql
# 客户端一次查询，精准聚合跨域数据
query GetUserDashboard {
  user(id: "123") {
    id
    name
    walletBalance
    orders(last: 5) {
      id
      amount
      status
    }
  }
}
```

### 3.1 GraphQL 的高维优势

- **免除接口编排**：前端直接接管数据组装权，新增页面无需后端配合加接口。
- **内建自描述 (Introspection)**：强类型的 Schema 机制使得文档与代码同源，利用 GraphiQL 等工具可以实现极致的联调体验。
- **渐进式演进**：无需像 REST 那样通过 `/v1`、`/v2` 升版，只需标记 `@deprecated` 废弃字段即可。

### 3.2 GraphQL 的工程挑战

- **N+1 数据库风暴**：因为查询是按图深度遍历解析的（Resolver），获取 1 个用户和 10 个订单，可能会触发 11 次 SQL 查询。**必须引入 `DataLoader` 等批处理/缓存中间件进行拦截聚合。**
- **AST 恶意查询**：客户端如果发送极深嵌套的查询（如用户查订单，订单查商品，商品再查用户），会瞬间击穿服务器 CPU。**必须在网关层进行 Query 深度限制和复杂度计算 (Complexity Assessment)。**
- **缓存失效**：由于绝大多数请求都是单一入口的 `POST /graphql`，CDN 和 HTTP 层的缓存全部失效，只能依赖客户端库（如 Apollo Client 的归一化缓存）或服务端 Redis 自建缓存。

## 4. RPC

RPC 抛弃了“**资源**”的包袱，回归程序员最熟悉的“**函数调用**”。现代前端接触最多的通常是轻量级的 JSON-RPC，或是基于 TypeScript 极致推导的 tRPC，而微服务底层则统治着基于 HTTP/2 的 gRPC。

```typescript
// tRPC 前端调用示例（自带极其严格的端到端类型推导）
const user = await trpc.user.getUserById.query({ id: '123' })
await trpc.order.payOrder.mutate({ orderId: '999', method: 'ALIPAY' })
```

### 4.1 RPC 的高维优势

- **行为表达绝对直接**：不再纠结 `PUT` 还是 `PATCH`，直接定义 `Login`、`ProcessRefund`。
- **性能天花板 (gRPC)**：采用 Protobuf 二进制序列化，体积比 JSON 小数倍，解析速度极快；底层强制 HTTP/2，支持双向流式通信。
- **端到端类型安全 (tRPC)**：在 Monorepo 架构中，前端直接导入后端的类型定义，修改后端接口，前端代码直接在编译时报错，彻底消灭联调期的类型对齐成本。

### 4.2 RPC 的代价

- **黑盒化与生态隔离**：无法直接用浏览器地址栏或 Postman 简单调试（gRPC 需要专门的 GUI 客户端如 BloomRPC，并通过 Envoy 等网关做 gRPC-Web 转换）。
- **版本灾难**：如果缺少严格的 Protobuf 字段约束规范，随意修改参数类型会导致严重的向下不兼容。

## 5. 工业级选型决策树

| 业务场景特征                                             | 推荐架构                     | 选型理由与补充策略                                                                    |
| -------------------------------------------------------- | ---------------------------- | ------------------------------------------------------------------------------------- |
| **公共开放 API (OpenAPI)** / 资源驱动型系统 / 强依赖 CDN | **REST**                     | 全球通用标准，接入成本极低。配合 Swagger/OpenAPI 生成规范。                           |
| **超复杂前台应用** / 多端视图差异巨大 / 聚合需求多       | **GraphQL**                  | 将后端微服务在网关层聚合成 Unified Graph，解放前端。                                  |
| **前后端同构 (Monorepo)** / 中后台管理系统 / 敏捷开发    | **tRPC / JSON-RPC**          | 极致的开发效率，跨栈类型推导，所见即所得。                                            |
| **微服务间内部通信** / 极高吞吐量 / 实时音视频信令       | **gRPC**                     | 压榨网络与 CPU 性能，二进制传输，支持双向 Streaming。                                 |
| **历史遗留微服务 + 灵活前台**                            | **BFF (REST/RPC) + GraphQL** | 用一个 Node.js/Go 中间层（BFF）封装底层的 RPC 接口，向前台暴露 GraphQL 供其灵活查询。 |

## 6. 前端关注点

### 6.1 状态管理与缓存策略的决裂

- **REST**：状态管理通常围绕全局 Store（Redux）或请求库（SWR / React Query）展开，缓存 Key 就是完整的 URL。
- **GraphQL**：前端必须引入庞大的生态（Apollo Client / Relay），其缓存机制基于**图和实体 ID (Normalized Cache)**。即无论通过哪个查询拿到 `User:123` 的数据，都会更新本地唯一的实体，实现跨组件状态自动同步。
- **RPC**：前端体验与本地异步函数无异，缓存完全依赖业务手动处理或结合 React Query 包装。

### 6.2 错误处理边界的模糊

- **REST**：高度依赖 HTTP 状态码体系（如 `401` 未授权，`404` 不存在，`422` 参数校验失败），网络层和业务层错误界限分明。
- **GraphQL**：无论业务是否报错，HTTP 状态码通常永远是 `200 OK`。错误信息被包裹在响应体的 `errors` 数组中，且可能存在**部分成功部分失败**的状态。
- **RPC (gRPC)**：拥有一套独立于 HTTP 的底层状态码体系（如 `UNAUTHENTICATED`, `DEADLINE_EXCEEDED`），前端拦截器需要做专门的映射翻译。

## 7. 常见问题 (FAQ) 与面试热点

### 7.1 POST 可以替代所有 HTTP 方法，为什么还要遵循 RESTful 规范？

纯用 `POST /api` 配合 Body 传参（本质是粗糙的 RPC 风格）完全能实现业务，但丧失了 Web 基础设施的红利：

- **阻断基础设施**：网关或 CDN 无法通过请求行判断操作意图，无法做到对 `GET` 自动缓存，对 `POST/PUT` 防重放攻击。
- **丢失语义校验**：丢失了 `PUT` 的幂等性语义，在网络抖动触发自动重试时极易引发数据脏写。

### 7.2 BFF (Backend For Frontend) 和 GraphQL 到底是什么关系？

BFF 是一种**架构思想**（为每一种前端定制一个专用后端胶水层），而 GraphQL 是一种**技术实现**。GraphQL 是实现 BFF 架构的最强武器：与其为 Web 端和 iOS 端分别写一个 BFF 接口，不如部署一个 GraphQL Gateway 作为统一 BFF，让多端客户端自己决定要什么字段。

### 7.3 GraphQL 是否会取代 REST？

不会。二者并非替代关系，而是互补关系。在底层微服务或领域驱动设计 (DDD) 中，依然以 REST 或 gRPC 为主干；而 GraphQL 越来越退化为一层“**数据聚合网关 (Data Graph Gateway)**”，部署在前端和微服务集群之间，专职处理视图层的数据组装脏活。
