# 微前端架构 (Micro Frontends) 全景解析

微前端是一种借鉴后端微服务思想的前端架构模式。它将庞大的单体应用（巨石应用）物理拆分为多个可**独立开发、独立构建、独立部署、独立运行**的小型子应用，最终在浏览器运行时由主应用（基座）进行无缝组合，为用户提供统一的交互体验。

## 1. 什么是微前端？

**微前端**是指将前端应用分解为多个更小的子应用，每个子应用独立开发、独立测试、独立部署，由**主应用（基座/容器）** 统一编排和整合。

> **💡 举个例子：**
> 一个电商平台包含：商品列表、购物车、个人中心、订单管理等功能模块。
>
> - **传统巨石应用：** 整个电商是一个项目，所有功能耦合在一起。10 人团队在同一仓库开发，合并冲突频繁，回归测试范围大，发布窗口长。
> - **微前端模式：** 商品团队维护"**商品子应用**"，订单团队维护"**订单子应用**"……每个子应用有独立仓库和 CI/CD，通过主应用路由整合，用户看到的仍是一个整体。

## 2. 核心价值与挑战

### 2.1 核心价值

- **工程解耦与自治**：多团队并行开发，拥有独立的迭代节奏和技术选型权，大幅降低跨部门沟通成本。
- **渐进式重构**：允许新老技术栈共存。老旧遗留系统无需一次性推翻，新业务可直接使用 Vue 3 等现代框架开发，平滑过渡。
- **独立容灾与回滚**：某个子应用发布引发线上 Bug，只需回滚该子应用，不影响整个中台系统的可用性。
- **按需极致加载**：首屏仅加载基座和当前路由命中的子应用资源，避免巨石应用首次加载时的大量无用代码解析。

### 2.2 核心挑战与应对策略

| 挑战领域            | 痛点描述                                                                  | 业界成熟解决方案                                                                                     |
| ------------------- | ------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------- |
| **环境隔离 (沙箱)** | 多个应用在同一个 `window` 下运行，极易产生全局变量污染和 CSS 样式冲突。   | JS 沙箱（Proxy 劫持 / iframe 物理隔离）；CSS 隔离（CSS Modules / Scoped / Shadow DOM）。             |
| **公共依赖冗余**    | 每个子应用都打包了一份 Vue/React，导致基础库重复下载，页面切换变卡。      | 依赖外置（CDN Externals）或基于底层构建工具的共享池（Module Federation `shared`）。                  |
| **通信复杂度**      | 子应用之间、基座与子应用之间的数据传递容易演变成“意大利面条”式的耦合。    | 保持极简通信。依赖 Props 透传配置，利用 URL 传参解耦，仅对 Token 等极少量全局状态使用 Shared Store。 |
| **路由抢占**        | 子应用内部的路由跳转可能与基座的全局路由规则发生冲突或覆盖。              | 基座担任路由总网关，严格为每个子应用分配特定的一级路由前缀（如 `/app-loan`）。                       |
| **运行态排查难**    | 发生白屏或抛出 `TypeError` 时，难以界定是基座的错误还是某个子应用的异常。 | 规范化日志，集成企业级 Sentry，并在不同子应用挂载时注入独有的分布式追踪打标。                        |

## 3. 主流实现方案与选型对比

微前端经过多年的演进，从最早的伪微前端（iframe），到基于路由的调度（single-spa），再到现在的开箱即用方案（qiankun、Wujie 等），生态已经非常成熟。

### 3.1 核心流派解析

| 方案                  | 特点                                    | 优点                                                     | 局限                                                         | 适用场景                                           |
| --------------------- | --------------------------------------- | -------------------------------------------------------- | ------------------------------------------------------------ | -------------------------------------------------- |
| **iframe**            | 浏览器原生隔离                          | 隔离最强、接入简单                                       | 体验割裂、通信复杂、性能开销大                               | 第三方系统嵌入、遗留系统集成                       |
| **qiankun**           | 基于 single-spa，HTML 入口 + Proxy 沙箱 | 开箱即用、国内生态成熟、解决历史包袱能力强               | **对 Vite ESM 产物支持较弱**，通常需借助 Rollup 插件强转格式 | 存量 Webpack 企业级系统快速落地                    |
| **Wujie (无界)**      | **Web Components + iframe**             | 极致的物理隔离，**原生完美兼容 Vite/ESM 构建**，极简接入 | iframe 存在一定的初始内存开销                                | 新旧系统混编、全线拥抱 Vite 的团队、高安全要求场景 |
| **Micro-app**         | Web Component + Proxy 沙箱              | 接入最简单、天然 JS/CSS 隔离、支持 Vite                  | 较新、社区较小、Shadow DOM 下的 UI 弹窗易逃逸                | 快速接入已有项目、中小团队                         |
| **Module Federation** | 运行时模块共享与动态加载                | 依赖复用率极高、无额外沙箱运行时损耗                     | 配置复杂、缺乏硬性物理隔离、工程化规约成本高                 | 统一技术栈的大型 SPA 拆分、组件级共享              |

### 3.2 选型策略决策树

| 业务场景现状                                                         | 推荐方案                   | 核心理由                                                                            |
| -------------------------------------------------------------------- | -------------------------- | ----------------------------------------------------------------------------------- |
| **包含大量新旧技术栈（jQuery/Vue2），且新业务重度依赖 Vite 构建**    | **Wujie (无界)**           | iframe 提供的物理级沙箱能完美兼容原生 ESM，无需改造 Vite 配置，旧系统也能严密隔离。 |
| **存量旧项目改造，主应用是 Webpack，追求国内最成熟的生态和避坑指南** | **qiankun**                | 社区案例最多，遇到边缘边界问题最容易搜索到现成解决方案。                            |
| **全新立项的企业级巨型中台，技术栈高度统一（如全线 Vue 3 + Vite）**  | **Vite Module Federation** | 抛弃沉重的运行时沙箱，追求极致的代码复用率和首屏渲染性能。                          |
| **仅仅是想在当前页面快速嵌入一个第三方黑盒系统**                     | **原生 iframe**            | 成本最低，安全性最高。                                                              |

## 4. 运行机制

一个成熟的微前端架构，主应用应保持轻量，不承载过多业务逻辑，避免重新变为"**巨石基座**"。：

```
用户访问 URL
    ↓
主应用（基座）：布局、菜单、认证、权限、路由分发
    ↓
子应用 A（Vue）  子应用 B（React）  子应用 C（Angular）
    ↓
基础设施：JS 沙箱、CSS 隔离、通信、预加载、监控、错误兜底
```

### 4.1 路由调度引擎

基座是整个应用的网关。它通过底层劫持浏览器的 `history.pushState` 和 `popstate` 事件，监听 URL 变化。根据 URL 前缀激活对应子应用，调度引擎挂起当前子应用，并动态获取下一个子应用的入口资源。

```
URL: /app-order/list     → 订单子应用处理 /list
URL: /app-goods/detail/1 → 商品子应用处理 /detail/1
```

### 4.2 标准状态机（生命周期）

无论哪种框架，子应用最终都需要被纳入基座的状态机管理：

- `bootstrap`：首次加载时触发，用于初始化全局字典、环境预热（全局仅一次）。
- `mount`：每次路由命中时触发。执行前端框架的实例化（如 Vue 的 `createApp().mount()`）。
- `unmount`：路由切走时触发。**这是微前端最关键的一环**，必须在此彻底销毁实例，解绑原生 `window.addEventListener`，清理定时器，否则会导致极其严重的内存泄漏。

:::details Vue 3 子应用的终极销毁规范

```js
let app = null

export async function mount(props) {
  app = createApp(App)
  app.mount(props.container)
}

export async function unmount(props) {
  // 1. 彻底卸载 Vue 实例，触发内部的 effectScope 销毁
  app?.unmount()
  app = null

  // 2. 必须手动清理挂载在 window 上的全局事件
  window.removeEventListener('resize', globalResizeHandler)

  // 3. 清理全局未闭环的副作用 (如 setInterval、未完成的 Axios 请求)
  clearAllTimers()
}
```

:::

### 4.3 通信机制

| 方式            | 适用场景                         | 注意事项                 |
| --------------- | -------------------------------- | ------------------------ |
| **Props 透传**  | 主应用向子应用传用户、权限、配置 | 简单可靠，适合单向传递   |
| **CustomEvent** | 低频事件通知                     | 需统一命名约定，及时解绑 |
| **共享 Store**  | 登录态、主题、语言等全局状态     | 避免业务状态过度共享     |
| **URL 参数**    | 筛选条件、详情 ID 等可序列化数据 | 适合可分享、可刷新的场景 |
| **接口通信**    | 业务数据交互                     | 优先通过后端 API 解耦    |

**推荐原则：** 能用 URL 和后端 API 解耦的，不强行做前端全局通信。

### 4.4 沙箱隔离机制边界

**CSS 隔离方案：**

| 方案                        | 原理               | 优点         | 局限                 |
| --------------------------- | ------------------ | ------------ | -------------------- |
| **CSS Modules**             | 构建时生成唯一类名 | 无运行时开销 | 第三方库样式难控制   |
| **BEM 命名**                | 规范类名前缀       | 零开销、简单 | 靠人为约束           |
| **postcss-prefix-selector** | 构建时自动加前缀   | 自动化、透明 | 可能影响选择器优先级 |
| **Shadow DOM**              | 浏览器原生隔离     | 最强隔离     | 弹窗/浮层会逃逸      |

**JS 沙箱方案：**

| 方案                         | 优点               | 局限                     |
| ---------------------------- | ------------------ | ------------------------ |
| **Proxy 沙箱**（如 qiankun） | 高性能、支持多实例 | 不兼容 IE                |
| **快照沙箱**                 | 兼容性好           | 性能开销大、不支持多实例 |
| **iframe**                   | 最强隔离           | 通信复杂、性能开销大     |

**子应用卸载必须清理：** window 事件、定时器、全局变量、订阅、WebSocket、未完成请求。

## 5. 常见问题与最佳实践

| 问题               | 建议                                                      |
| ------------------ | --------------------------------------------------------- |
| **样式污染**       | 统一应用前缀，使用 CSS Modules 或构建时前缀               |
| **弹窗层级异常**   | 统一 Modal 挂载容器，注意 Shadow DOM 下的 Portal/Teleport |
| **依赖重复加载**   | 对 React、Vue、组件库等配置 shared 或 external            |
| **刷新 404**       | 服务端 fallback 到主应用入口                              |
| **跨应用跳转混乱** | 主应用提供统一导航 API                                    |
| **全局状态滥用**   | 只共享认证、权限、主题和语言等公共状态                    |
| **卸载不干净**     | unmount 中清理事件、定时器、订阅和全局副作用              |
| **线上排查困难**   | 统一日志、错误监控、性能监控和版本标识                    |

## 6. 性能优化与高可用兜底

- **按需加载**：仅随路由按需加载，不冗余。
- **智能预热**：利用空闲/悬停间隙 (`requestIdleCallback`)，提前预载高频子应用。
- **依赖共享**：抽离核心框架（Vue/React），通过 CDN 或 Module Federation 杜绝重复打包。
- **分级缓存**：入口 `index.html` 走短缓存，静态资产 (JS/CSS) 走哈希长缓存。
- **异常兜底**：子应用加载失败时，基座提供标准骨架屏加自动重试，告别白屏死机。
- **极简通信**：砍掉高频的跨端 EventBus 事件，复杂状态交由后端 API 解耦。

## 7. 微前端 vs. Monorepo

微前端关注**运行时组合**，Monorepo 关注**代码仓库组织**，两者互补而非对立。

| 维度           | Monorepo                    | 微前端                     |
| -------------- | --------------------------- | -------------------------- |
| **关注阶段**   | 开发期、构建期              | 运行期、部署期             |
| **核心目标**   | 统一代码管理和依赖复用      | 独立运行部署和运行时组合   |
| **技术栈**     | 通常统一                    | 可以不同                   |
| **代码共享**   | workspace 协议天然支持      | 需额外设计 shared 机制     |
| **运行时隔离** | 无                          | 需要沙箱和样式隔离         |
| **常用工具**   | pnpm、Turborepo、Changesets | qiankun、Module Federation |

> **实践建议：** Monorepo 管理子应用仓库和公共包，微前端方案负责运行时的组合与隔离，两者结合能充分发挥各自优势。

```
micro-frontend-monorepo/
├── apps/
│   ├── container/        # 主应用 (Monorepo 管理)
│   ├── app-order/        # 订单子应用 (Monorepo 管理 + 微前端暴露)
│   └── app-goods/        # 商品子应用
├── packages/
│   ├── shared-ui/        # 公共 UI 组件库
│   ├── shared-utils/     # 公共工具函数
│   └── eslint-config/    # 共享 ESLint 配置
├── pnpm-workspace.yaml
├── turbo.json
└── package.json
```

## 8. 落地实施与评估清单

在决定上马微前端架构前，请团队对照以下条件进行审视。如果大部分答案为否，建议暂缓引入，先夯实工程化基建：

- **业务边界是否清晰？**（如果两个模块之间需要疯狂地跨应用传递海量数据，说明不应强行拆分）。
- **基建完备度是否达标？** 是否具备成熟的自动化部署（CI/CD）体系，支撑多个子应用的高效独立发版？
- **规范是否达成共识？** 是否制定了明确的子应用接入规范、路由前缀分配机制以及统一的 UI 交互视觉标准？
- **监控体系是否建立？** 是否具备全局统筹的错误捕获与日志监控系统，以便在分布式架构中快速定位线上故障？
