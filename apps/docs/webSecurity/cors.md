# CORS (Cross-Origin Resource Sharing) 跨域资源共享

在 Web 开发和前后端分离架构中，**CORS（跨域资源共享）** 是前端工程师和后端架构师最常遇到的网络安全与通信协商机制，也是高级面试中的必考题。虽然它经常以控制台那串刺眼的红色报错形式出现，但它的**设计初衷、分类判定和防御配置策略**有着极其严谨的安全逻辑。

## 1. 简单请求 (Simple Requests)

**核心本质**：请求完全符合传统的 Web 表单提交特征，浏览器认为此类操作是“**低风险**”的，因此直接发送真实的 HTTP 请求，并在收到响应后校验跨域放行权限。

**攻击目标**：绕过浏览器的同源策略限制以获取敏感数据。

**一句话理解**：**“先斩后奏：请求直接发给服务器，如果没有跨域权限，浏览器当场扣留响应结果。”**

### 1.1 简单请求的触发条件 (必须同时满足)

| 限制维度         | 严格要求                                                                                                                                                                  |
| ---------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **HTTP 方法**    | 只能是 `GET`、`POST`、`HEAD` 三者之一。                                                                                                                                   |
| **HTTP 请求头**  | 只能包含安全的默认系统头（如 `Accept`、`Accept-Language`、`Content-Language`、`Last-Modified` 等）。**绝对不允许**包含任何自定义请求头（如 `Authorization`、`X-Token`）。 |
| **Content-Type** | 只能是以下三种传统表单格式之一：<br/>1. `text/plain`<br/>2. `multipart/form-data` (常用于文件上传)<br/>3. `application/x-www-form-urlencoded` (传统的键值对表单)          |

### 1.2 简单请求的工作流程

- 前端发起请求，浏览器在网络层检测到该请求符合简单请求的物理特征。
- **浏览器直接放行**：浏览器不进行事前询问，直接将真实请求发送给服务器，并在请求头中自动带上来源标识 `Origin: http://test.com`。
- **服务器真实执行**：服务器接收到请求，**正常执行底层的业务逻辑**（如数据库的增删改查），并根据自身的跨域配置在响应头中返回 `Access-Control-Allow-Origin`。
- **浏览器的“事后”拦截**：浏览器接收到服务器的响应报文，开始核对响应头。如果响应头中没有 `Access-Control-Allow-Origin`，或者该字段的值与当前站点的域名不匹配，浏览器就会在控制台抛出 CORS 错误，并**拦截响应体，绝对不把数据交给前端 JavaScript 代码**。

## 2. 预检请求 (Preflight Requests)

**核心本质**：当请求可能对服务器数据产生不可逆的影响，或者使用了现代 Web 的复杂数据格式（如 JSON）时，浏览器为了安全起见，必须先发送一个探测性质的请求去“**探路**”，征得服务器的显式同意后，才发送真正的业务请求。

**攻击目标**：保护服务端免受未经授权的复杂跨域指令（如 `DELETE`、`PUT`）的破坏。

**一句话理解**：**“投石问路：先发个 OPTIONS 请求问问服务器行不行，服务器说行，浏览器才发真实的业务请求。”**

### 2.1 预检请求的触发条件 (满足其一即可)

- 使用了 `PUT`、`DELETE`、`PATCH` 等非简单 HTTP 方法。
- 请求体的 `Content-Type` 使用了 `application/json` 格式（现代前端主流的 API 交互格式）。
- 携带了自定义的 HTTP 请求头（如跨域状态下常见的 `Authorization: Bearer <token>`、`X-Requested-With`）。

### 2.2 预检请求的工作流程

- 前端发起复杂的 AJAX/Fetch 请求（例如一个带有 JSON 载荷的 `POST` 请求）。
- 浏览器拦截该请求，**暂停发送真实数据**，转而向同一个目标 URL 发送一个 **HTTP 方法为 `OPTIONS`** 的预检请求。
  - 携带 `Access-Control-Request-Method`（声明接下来的真实请求会使用什么方法）。
  - 携带 `Access-Control-Request-Headers`（声明接下来的真实请求会携带哪些自定义头）。
- 服务器接收到 `OPTIONS` 请求，网关或框架中间件进行权限校验。如果同意跨域，则返回一个状态码为 204（No Content）或 200 的响应，并在响应头中附带：
  - `Access-Control-Allow-Origin` (允许的源)
  - `Access-Control-Allow-Methods` (允许的方法白名单)
  - `Access-Control-Allow-Headers` (允许的自定义头白名单)
- 浏览器收到并验证 `OPTIONS` 响应通过后，**才真正发起**那条扣留下的业务请求。如果预检失败，真实的业务请求根本不会离开浏览器。

## 3. 身份凭证与预检缓存 (防线与优化)

### 3.1 身份凭证 (Credentials)

默认情况下，跨域的 CORS 请求是**完全匿名**的，浏览器发起请求时绝对不会自动携带目标域名的 Cookie 或 HTTP 认证信息。要打通这层壁垒，需要前后端双向配置：

- **防线一：前端显式声明开启凭证开关**。
  - Fetch API: 设置 `credentials: 'include'`。
  - Axios / XMLHttpRequest: 设置 `withCredentials: true`。

- **防线二：后端显式返回放行许可头**。
  - 服务端响应中必须包含 `Access-Control-Allow-Credentials: true`。

- **警告**：一旦开启了身份凭证（`Allow-Credentials: true`），后端的 `Access-Control-Allow-Origin` **绝对不能**设置为通配符 `*`。服务端必须动态读取请求头中的 `Origin` 字段，并将其原封不动地反射回响应头中，否则浏览器会直接报错拦截。

### 3.2 预检缓存 (Access-Control-Max-Age)

- **核心痛点**：每次发送 JSON 数据的 API 请求都要经历 `OPTIONS + POST` 两次网络往返，这不仅让系统的网络请求数量瞬间翻倍，更极大延长了前端的页面网络延迟（TTFB），严重消耗服务器吞吐量。
- **优化手段**：服务器在 `OPTIONS` 的成功响应头中，加上 `Access-Control-Max-Age: <秒数>`（例如 `86400`，代表 24 小时）。
- **优化效果**：浏览器会在指定的秒数内“**记住**”这次审批结果。在缓存有效期内，相同特征的复杂跨域请求将直接绕过 OPTIONS 探测阶段，变回一次网络往返的丝滑体验。

## 4. CORS 深度对比及常见问题 (FAQ)

### 4.1 简单请求 vs 预检请求 核心差异对比

| 对比维度         | 简单请求 (Simple Request)                                                         | 预检请求 (Preflight Request)                                                                           |
| ---------------- | --------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------ |
| **请求网络次数** | 1 次（直接发送真实的业务请求）。                                                  | 2 次（1 次 OPTIONS 预检探测 + 1 次真实业务请求）。                                                     |
| **典型触发场景** | 传统的 HTML `<form>` 表单提交、纯文本 POST 请求。                                 | 现代前端开发中使用 Axios 发送 JSON 数据、请求头携带 JWT Token、使用 `PUT`/`DELETE` 执行 RESTful 操作。 |
| **安全拦截时机** | **事后拦截**。请求已经抵达服务器并被执行，只是浏览器拒绝将返回的结果交给前端 JS。 | **事前拦截**。如果 OPTIONS 预检失败，真正的业务请求根本不会被发出，从源头上保护了服务器数据。          |

### 4.2 为什么我在前端配置了代理 (Proxy) 之后，就不存在跨域问题了？

- **根本原因**：跨域同源策略（Same-Origin Policy）是**浏览器**独有的安全限制。
- **运行机制**：在开发阶段，由 Vite 或 Webpack 在本地启动了一个同源的 Node.js 中间件服务器。前端的代码直接请求这个本地服务器（同源，不触发 CORS）。本地服务器收到请求后，在后台通过纯代码将请求转发给真正的后端目标服务器。因为**服务器与服务器之间的通信不受浏览器同源策略限制**，因此完美绕过了跨域报错。

### 4.3 CORS 跨域报错，到底是前端的锅还是后端的锅？

- **99% 的情况下，是后端（或运维网关）的锅。**
- 跨域的本质是浏览器代替前端去询问服务器“**是否愿意接受外来域名的访问**”。前端除了在需要携带 Cookie 时配置一下 `withCredentials` 外，没有权限改变任何 HTTP 响应头。真正决定跨域能否成功的许可权（即返回各类 `Access-Control-Allow-*` 响应头），完全掌握在服务端的 Nginx 配置、网关或后端的 CORS 中间件手里。

### 4.4 为什么抓包看到后端明明返回了 200，但前端控制台依然报 CORS 错误？

- **场景 A（简单请求）**：后端虽然业务逻辑执行成功并成功返回了状态码 200，但其响应头里缺失了 `Access-Control-Allow-Origin` 字段，被浏览器在最后一关无情拦截。
- **场景 B（预检请求）**：后端没有针对 `OPTIONS` 方法做专门的路由放行或中间件拦截。虽然业务路由对 `POST` 正常响应 200，但面对前置的 `OPTIONS` 请求时，后端可能报错或者返回了没有跨域头的 200，导致预检死在第一步，真实的 `POST` 请求被浏览器直接扼杀。

### 4.5 为什么现在的跨域请求中，几乎都不带 withCredentials 参数了？

- **因为现代前后端架构的主流鉴权范式，已经从“依赖浏览器自动发送 Cookie” 全面转向了“前端手动携带 JWT (Token)”。**
- 过去使用 Session/Cookie 鉴权时，必须开启 `withCredentials` 来强制浏览器在跨域时带上身份凭证。但在现代微服务架构中，前端通常会将登录后获取的 JWT 存储在 `localStorage` 中，并在发送请求时通过 Axios 拦截器**手动**将其塞入请求头（如 `Authorization: Bearer <token>`）。既然身份信息是前端用 JS 代码主动塞进 HTTP Header 的，不再依赖浏览器底层的 Cookie 自动发送机制，`withCredentials` 自然就失去了存在的意义。此外，抛弃 Cookie 转向手动携带 Token 的做法，还能从物理层面上彻底免疫 CSRF（跨站请求伪造）攻击。
