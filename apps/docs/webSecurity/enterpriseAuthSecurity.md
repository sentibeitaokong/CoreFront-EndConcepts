# 企业级鉴权与安全方案

企业级鉴权不是“登录 + 菜单权限”，而是一套围绕身份、会话、权限、接口、前端暴露面、审计和风控的闭环。目标是：**身份可信、权限可控、凭证可撤销、数据可隔离、操作可审计、风险可收敛**。

## 1. 总体分层

| 层级   | 解决问题           | 关键能力                                                 |
| :----- | :----------------- | :------------------------------------------------------- |
| 身份层 | 用户是谁           | SSO、OIDC、SAML、LDAP、MFA、账号生命周期                 |
| 会话层 | 登录态怎么维持     | Session、Access Token、Refresh Token、设备管理、统一登出 |
| 权限层 | 用户能做什么       | RBAC、ABAC、菜单权限、按钮权限、接口权限、数据权限       |
| 租户层 | 数据属于谁         | `tenantId` 隔离、组织隔离、资源归属、跨租户审计          |
| 接口层 | 请求是否可信       | CSRF、CORS、签名、防重放、限流、幂等、最小返回           |
| 前端层 | 如何减少暴露面     | 路由守卫、权限组件、CSP、敏感信息脱敏、Source Map 私有化 |
| 工程层 | 代码和依赖是否可信 | SAST、依赖审计、Secret 扫描、CI Gate、灰度发布           |
| 审计层 | 出事后能否追溯     | 登录日志、操作日志、风控告警、Trace ID、会话吊销记录     |

核心原则：

- 前端负责体验控制和暴露面收敛，不能作为最终安全边界。
- 服务端负责最终鉴权、数据过滤、会话吊销和审计落库。
- 身份平台负责统一认证、MFA、账号生命周期和企业 SSO。
- 高风险操作要按风险等级增加二次认证、审批或风控拦截。

## 2. 推荐架构

### 2.1 企业 Web 首选链路

```text
Browser
  -> Web App
  -> SSO / IdP
  -> App Backend / BFF
  -> Business APIs
```

推荐流程：

1. 用户访问业务系统，未登录时跳转统一身份平台。
2. IdP 完成密码、MFA、企业微信、LDAP 或其他认证。
3. 业务系统使用 OIDC Authorization Code + PKCE 接收回调。
4. App Backend / BFF 校验 `state`、`nonce`，用 `code` 换 Token。
5. 后端校验 `id_token` 的签名、`iss`、`aud`、`exp`、`nonce`。
6. 后端创建本系统会话，给浏览器下发 `HttpOnly; Secure; SameSite=Lax` Cookie。
7. 后续接口由后端根据会话、用户、角色、组织、租户和数据范围做鉴权。

更推荐让后端参与登录回调和 Token 交换。第三方身份 Token 留在服务端，前端只持有应用自己的短期登录态或 HttpOnly Cookie。

### 2.2 BFF 的价值

BFF（Backend For Frontend）适合企业后台、微前端和多业务系统：

- 隐藏 IdP Token、Refresh Token、内部 API Token，不暴露给浏览器 JS。
- 聚合用户信息、菜单树、权限点、租户信息，减少前端拼装安全上下文。
- 统一处理 CSRF、CORS、Cookie、会话续期、登出和 401 / 403。
- 把内部服务鉴权、字段裁剪、数据脱敏放在服务端完成。

不建议前端直接长期保存第三方 `access_token` 或 `refresh_token`。前端包、localStorage、IndexedDB、调试工具、浏览器插件和第三方脚本都不是长期密钥的安全存储位置。

## 3. 凭证与会话策略

| 凭证            | 生命周期     | 存储位置                       | 要点                                  |
| :-------------- | :----------- | :----------------------------- | :------------------------------------ |
| Session ID      | 中短期       | HttpOnly Cookie                | 企业 Web 首选，服务端可主动失效       |
| Access Token    | 短期         | 内存 / Header / Cookie         | 建议 5 到 15 分钟，权限变化时尽快失效 |
| Refresh Token   | 中长期       | HttpOnly Cookie / 后端安全存储 | 必须轮换、绑定设备、支持吊销          |
| CSRF Token      | 页面或会话级 | Header / 页面变量              | 写操作校验，不能只依赖 SameSite       |
| One-time Ticket | 极短期       | 后端一次性消费                 | 登录回调、下载、敏感跳转可用          |

基线策略：

- Web 管理后台优先使用 `Session + HttpOnly Cookie`。
- Cookie 使用 `HttpOnly; Secure; SameSite=Lax; Path=/`，能不设置 `Domain` 就不设置。
- 跨站 iframe、跨站前后端分离或第三方登录需要 `SameSite=None; Secure` 时，必须加强 CSRF Token 和 `Origin` 校验。
- Session 设置空闲过期时间和绝对过期时间，例如 30 分钟空闲过期、8 小时绝对过期。
- Refresh Token 使用轮换机制：每次刷新发新令牌，旧令牌立即失效；旧令牌再次出现时吊销整个会话族。
- 修改密码、重置 MFA、禁用账号、移除角色、离职、退出企业后，服务端主动吊销相关会话和刷新令牌。

## 4. 权限模型

### 4.1 RBAC：大多数后台的起点

```text
用户 -> 角色 -> 权限点 -> 菜单 / 页面 / 按钮 / API
```

常见权限点：

| 类型     | 示例               | 前端作用       | 后端要求              |
| :------- | :----------------- | :------------- | :-------------------- |
| 菜单权限 | `order`            | 控制导航展示   | 不能替代接口鉴权      |
| 页面权限 | `order:list`       | 控制路由进入   | 直接访问 URL 仍要校验 |
| 操作权限 | `order:delete`     | 控制按钮展示   | 写接口必须再次鉴权    |
| 接口权限 | `api:order:delete` | 通常不暴露细节 | 网关或服务端强制校验  |

前端隐藏按钮只是体验优化。删除、导出、审批、付款、改密、授权这类操作，即使前端有按钮权限，服务端也必须基于当前会话重新判断。

### 4.2 ABAC：复杂组织和高风险业务

ABAC 根据用户、资源、环境、动作一起判断，适合组织层级复杂、数据范围复杂、风险要求高的系统。

| 属性     | 示例                                           |
| :------- | :--------------------------------------------- |
| 用户属性 | 部门、岗位、职级、地区、雇佣状态、MFA 状态     |
| 资源属性 | 订单所属部门、客户负责人、数据密级、租户、项目 |
| 环境属性 | IP、设备、地理位置、时间、网络类型、风险等级   |
| 动作属性 | 查看、编辑、删除、导出、审批、授权             |

规则例子：

- 销售只能查看自己负责的客户。
- 部门经理可以查看本部门和子部门数据。
- 财务导出报表需要 MFA 二次确认。
- 非公司网络访问敏感页面时只能只读。
- 高密级数据只允许受管设备访问，并禁止批量导出。

### 4.3 数据权限和租户隔离

数据权限必须在服务端执行，不能只依赖前端过滤。

常见数据范围：

- 本人数据。
- 本部门数据。
- 本部门及子部门数据。
- 指定项目、区域、租户、客户或资源组。
- 全部数据。

多租户 SaaS 必须额外注意：

- 所有业务数据带 `tenantId` 或等价隔离字段。
- 服务端从登录上下文确定租户，不能信任前端传入的 `tenantId`。
- 查询、更新、删除都要自动追加租户条件和数据范围条件。
- 跨租户管理员操作必须走独立权限、二次确认和审计。
- 缓存 Key、消息 Topic、文件路径、搜索索引也要带租户隔离维度。

## 5. 前端落地

### 5.1 路由和初始化

前端路由守卫只负责体验闭环：

- 未登录跳转登录页或触发 SSO。
- 登录回调页只处理 `code`、`state`，尽快交给后端。
- 应用启动先拉取 `/me`、菜单树、权限点、租户列表，再渲染业务页面。
- 无页面权限跳转 403；会话失效跳转登录；租户无权访问提示切换租户。
- 动态路由可以由权限生成，但服务端仍要拦截直接访问接口的请求。

### 5.2 权限组件和指令

常见封装：

```text
v-permission="'order:delete'"
<Permission code="order:export">...</Permission>
```

注意事项：

- 权限组件只控制展示，不代表接口安全。
- 高风险按钮点击后仍要由接口二次鉴权。
- 批量导出、审批、付款、改密、授权建议加二次确认、MFA 或审批流。
- 权限点应使用稳定编码，不要把中文菜单名、按钮文案当权限标识。

### 5.3 前端缓存边界

可以缓存：

- 昵称、头像、当前租户、菜单树、权限点列表。
- UI 偏好、最近访问页面、低敏筛选条件。

不要缓存：

- refresh token、长期 access token、client secret、私钥。
- 完整身份证、银行卡、手机号明文、邮箱验证码。
- 管理员后门权限、风控标签、服务端内部策略。
- 任何被篡改后会影响服务端鉴权的字段。

本地缓存只适合提升体验。服务端必须把前端传入的用户、角色、部门、租户、数据范围都视为不可信输入。

## 6. 接口安全

| 风险            | 防护重点                                        |
| :-------------- | :---------------------------------------------- |
| CSRF            | SameSite、CSRF Token、`Origin` / `Referer` 校验 |
| XSS 后借用身份  | CSP、输入输出转义、敏感操作二次确认、最小权限   |
| 水平 / 垂直越权 | 服务端 RBAC / ABAC / 数据权限 / 租户隔离        |
| 重放请求        | nonce、时间戳、幂等键、签名、一次性票据         |
| 暴力请求        | 限流、验证码、账号锁定、IP / 设备风控           |
| 跨域误配        | 精确 CORS 白名单，不使用 `*` + credentials      |
| 数据泄露        | 字段裁剪、脱敏、分页限制、导出审批、水印        |

接口基线：

- 所有权限判断都在服务端执行。
- 敏感操作不用 GET，写操作必须具备 CSRF 防护。
- 接口返回最小字段集合，默认不返回内部状态、风控标签和密级字段。
- 管理接口、用户接口、开放 API 分离权限域。
- 批量查询、导出、下载要限制数量、频率和字段。
- 所有写操作、高风险读操作、权限变更都记录审计日志。
- 401 表示未登录或会话失效，403 表示已登录但无权限，前端要区分处理。

## 7. 风险分级

不同操作不应使用同一强度的校验。

| 风险级别 | 操作示例                           | 推荐控制                             |
| :------- | :--------------------------------- | :----------------------------------- |
| 低风险   | 查看普通列表、修改 UI 偏好         | 登录态 + 基础权限                    |
| 中风险   | 编辑订单、修改资料、提交审批       | 服务端鉴权 + 审计                    |
| 高风险   | 删除、批量导出、付款、授权、改密   | 二次确认 + MFA / 短时操作票据 + 审计 |
| 极高风险 | 跨租户操作、权限模型变更、关闭风控 | 双人审批 + 强 MFA + 告警 + 回滚预案  |

二次认证不一定每次都要求重新登录，可以使用短时效操作票据：

```text
用户触发高风险操作
  -> 后端返回 need_mfa
  -> 用户完成 MFA
  -> 后端签发 5 分钟一次性 action_ticket
  -> 前端携带 ticket 重试操作
  -> 后端消费 ticket 并写审计
```

## 8. 前端安全基线

| 能力         | 建议配置                                                 |
| :----------- | :------------------------------------------------------- |
| HTTPS        | 全站强制 HTTPS，启用 HSTS                                |
| Cookie       | `HttpOnly; Secure; SameSite=Lax`                         |
| CSP          | 限制脚本、样式、图片、接口外连来源，减少 `unsafe-inline` |
| SRI          | 第三方 CDN 脚本固定版本并校验完整性                      |
| ClickJacking | 使用 `frame-ancestors` 限制可嵌入来源                    |
| Source Map   | 上传监控平台后不公开部署                                 |
| 日志         | Token、Cookie、密码、验证码、PII 脱敏                    |
| 依赖         | lockfile、依赖审计、限制 install script、定期升级        |
| 构建         | Secret 扫描、SAST、类型检查、测试、CI Gate               |

前端项目应把这些配置沉淀成模板和流水线规则，避免每个业务线临时配置。

## 9. 审计与监控

企业鉴权系统必须能回答：谁在什么时候、从哪里、使用什么设备、对什么资源、做了什么、结果如何。

需要记录：

- 登录成功 / 失败、MFA 成功 / 失败、账号锁定。
- Token 刷新、退出登录、统一登出、会话吊销、设备解绑。
- 权限变更、角色变更、组织变更、租户切换。
- 敏感操作：导出、删除、审批、付款、改密、授权。
- 异常行为：异地登录、短时间大量失败、异常导出、越权访问、接口爆破。

日志字段建议：

| 字段                          | 说明                 |
| :---------------------------- | :------------------- |
| `traceId`                     | 请求链路 ID          |
| `userId`                      | 操作用户             |
| `tenantId`                    | 租户或组织           |
| `sessionId`                   | 会话标识，建议存摘要 |
| `action`                      | 操作类型             |
| `resourceType` / `resourceId` | 资源对象             |
| `ip` / `deviceId`             | 来源信息             |
| `riskLevel`                   | 风险等级             |
| `result`                      | 成功、失败、拦截     |
| `reason`                      | 失败原因或风控原因   |

日志中不要保存明文密码、完整 Token、完整 Cookie、验证码和完整敏感个人信息。

## 10. 落地清单

| 阶段       | 必做项                                                            |
| :--------- | :---------------------------------------------------------------- |
| 登录接入   | OIDC Authorization Code + PKCE，校验 `state`、`nonce`、回调白名单 |
| 会话管理   | HttpOnly Cookie、服务端 Session、空闲过期、绝对过期、主动吊销     |
| Token 治理 | Access Token 短时效，Refresh Token 轮换、设备绑定、异常复用检测   |
| 权限模型   | RBAC 起步，复杂场景补 ABAC、数据权限、租户隔离                    |
| 前端实现   | 动态路由、权限组件、403 页面、刷新后权限恢复、401 / 403 分流      |
| 接口防护   | 服务端鉴权、CSRF、防重放、限流、幂等、最小字段返回                |
| 高危操作   | 二次确认、MFA、短时操作票据、审批流、审计和告警                   |
| 工程治理   | CSP、Source Map 私有上传、依赖审计、Secret 扫描、CI Gate          |
| 监控审计   | 登录日志、操作日志、异常告警、Trace ID、会话吊销记录              |

## 11. 推荐组合

普通企业后台：

- OIDC SSO 登录。
- 后端换 Token 并校验 `id_token`。
- `Session + HttpOnly Cookie` 维持登录态。
- RBAC 控制菜单、页面、按钮、接口。
- 服务端执行数据权限和租户隔离。
- SameSite + CSRF Token + `Origin` 校验保护写操作。
- 登录、权限变更、敏感操作进入审计。

高安全系统：

- 登录强制 MFA，异常登录触发额外验证。
- Access Token 短时效，Refresh Token 轮换并绑定设备。
- 高风险操作使用 MFA、短时操作票据或双人审批。
- 高敏页面减少第三方脚本，严格 CSP 和字段脱敏。
- 导出、审批、付款、授权进入风控、审计和告警。

多租户 SaaS：

- 所有数据、缓存、文件、消息和搜索索引带租户隔离维度。
- 服务端从登录上下文确定租户，不信任前端传入租户。
- RBAC + ABAC + 数据范围 + 资源归属共同鉴权。
- 跨租户管理员操作必须独立授权、二次确认和审计。

结论：**前端负责让用户看到正确的入口并减少暴露面，后端负责最终鉴权、数据隔离和审计闭环，身份平台负责统一认证、MFA 和会话治理。**
