# 敏感信息泄露与供应链安全

前端安全不只包括 XSS、CSRF、跨域，还包括两个工程化风险：**敏感信息被打进前端产物**，以及**依赖供应链被污染**。

## 1. 核心原则

任何进入浏览器的代码、Source Map、静态资源、接口响应，都默认可以被用户查看、复制和分析。

一句话原则：**前端没有真正的秘密。**

因此前端只能保存公开配置、短期凭证或经过最小化处理的数据，不能保存服务端密钥、签名私钥、长期 Token、数据库密码、云厂商 Secret Key。

## 2. 常见泄露面

| 位置         | 风险             | 典型问题                           |
| :----------- | :--------------- | :--------------------------------- |
| `.env` 变量  | 构建时注入前端包 | 把 Secret 当公开配置               |
| JS Bundle    | 所有人可下载分析 | 写死 Token、内部地址、测试账号     |
| Source Map   | 还原源码结构     | 暴露注释、接口路径、业务规则       |
| HTML 注释    | 直接可见         | 暴露调试信息、账号、环境说明       |
| 接口响应     | 前端可抓包       | 返回过多用户字段或权限字段         |
| LocalStorage | JS 可读          | Token、用户信息被 XSS 读取         |
| 错误日志     | 平台长期保存     | 打印 Authorization、Cookie、手机号 |
| 第三方脚本   | 运行在页面主线程 | 可读取页面数据、发起外连           |

## 3. `.env` 与构建时注入

很多框架会把特定前缀的环境变量注入到前端代码，例如 Vite 的 `VITE_`。

```js
VITE_API_BASE_URL=https://api.example.com
VITE_PUBLIC_KEY=xxx
```

这些变量只要被前端使用，就会进入最终产物。它们适合放公开配置，不适合放密钥。

不要放：

- 数据库密码。
- JWT 签名 secret。
- OAuth `client_secret`。
- 云厂商 Secret Key。
- 私有 API Token。
- 内部系统管理员账号。

可以放：

- API Base URL。
- 公开的监控 DSN。
- OAuth `client_id`。
- 静态资源域名。
- 前端功能开关。

### 3.1 环境变量治理建议

- 明确区分“**构建时变量**”和“**服务端运行时变量**”。
- 前端变量使用白名单前缀，例如 `VITE_PUBLIC_`，减少误用。
- CI 中扫描产物，发现 `secret`、`token`、`password`、`private_key` 等关键词时阻断发布。
- 后端密钥只保存在服务端运行环境或 Secret Manager，不进入前端构建流程。

## 4. Source Map 与构建产物

Source Map 能把压缩后的 JS 映射回源码，方便调试，但在线上公开 Source Map 会暴露：

- 源码结构。
- 注释中的内部说明。
- 未导出的工具函数。
- 接口路径和业务规则。
- 可能被误写进源码的敏感信息。

建议：

- 生产环境不要公开可访问的 `.map` 文件。
- 如果错误监控需要 Source Map，上传到监控平台后不要部署到公网。
- CI 中扫描 `dist`，避免 `.map`、测试文件、临时文件被一起发布。
- 构建产物发布前做敏感关键词扫描和依赖清单归档。

## 5. 接口响应与日志脱敏

前端只展示昵称，不代表接口可以返回完整用户对象。

错误示例：

```json
{
  "id": 1,
  "name": "Alice",
  "phone": "13800000000",
  "idCard": "110101199001011234",
  "role": "admin",
  "passwordHash": "..."
}
```

接口原则：

- 当前页面只需要什么字段，就只返回什么字段。
- 权限判断必须在服务端执行，前端只做体验优化。
- 手机号、邮箱、证件号、银行卡、地址等敏感字段按场景脱敏。
- 管理后台接口也要按角色过滤字段，不能默认返回全量对象。
- 不要把 `passwordHash`、盐值、内部权限表达式、风控标签返回给前端。

日志原则：

- 不打印 `Authorization` Header、Cookie、完整 Token。
- 不打印完整用户对象、订单对象、支付对象。
- 错误监控上报前做字段过滤和脱敏。
- 日志平台本身也要有访问控制、保留期限和审计记录。

## 6. 浏览器存储风险

| 存储位置       | 风险                                 | 建议                                    |
| :------------- | :----------------------------------- | :-------------------------------------- |
| localStorage   | 持久化且 JS 可读，XSS 可直接窃取     | 不放长期高权限 Token                    |
| sessionStorage | 页面会话级 JS 可读                   | 只放低敏短期状态                        |
| IndexedDB      | 容量大，适合业务缓存，也可被 JS 读取 | 敏感数据加密或不落盘                    |
| Cookie         | 可自动携带，可能引发 CSRF            | 登录态使用 `HttpOnly; Secure; SameSite` |
| 内存变量       | 刷新丢失，XSS 当前会话仍可读         | 适合短期 access token                   |

不要因为 localStorage 使用方便，就把长期 refresh token、管理员权限、完整用户画像放进去。

## 7. 供应链安全

现代前端依赖大量 npm 包，只要其中一个依赖被投毒、劫持或误装，恶意代码就可能进入构建和运行环境。

| 风险                  | 说明                       |
| :-------------------- | :------------------------- |
| Typosquatting         | 安装了拼写相近的恶意包     |
| Dependency Confusion  | 内部包名被公网同名包抢占   |
| Maintainer 账号被盗   | 正常包发布了恶意版本       |
| Install Script        | `postinstall` 执行恶意命令 |
| Transitive Dependency | 间接依赖被污染             |
| CDN 第三方脚本        | 运行时加载的脚本被篡改     |

## 8. npm 依赖治理

### 8.1 锁定依赖版本

必须提交 lockfile：

- `package-lock.json`
- `pnpm-lock.yaml`
- `yarn.lock`

CI 使用 frozen lockfile，保证开发机、CI、生产构建使用一致的依赖解析结果。

### 8.2 谨慎升级依赖

升级依赖前关注：

- 是否是主版本升级。
- changelog 是否包含破坏性变更。
- 包的维护者、仓库、下载量是否异常。
- 是否突然新增大量依赖。
- 是否新增 `postinstall`、`preinstall` 等脚本。
- 是否出现异常混淆代码、远程下载二进制、访问环境变量等行为。

### 8.3 限制安装脚本

部分恶意包会通过安装脚本执行命令。

可以在 CI 中使用：

```bash
pnpm install --ignore-scripts
```

但要注意，有些正常包确实依赖安装脚本构建原生模块，需要结合项目情况评估。更稳妥的方式是：默认限制脚本，允许列表中放行必要依赖。

### 8.4 做依赖审计

常见工具：

```bash
pnpm audit
npm audit
```

审计不是银弹，它主要发现已公开漏洞。对新投毒、账号被盗、业务误用仍需要人工 review。

## 9. 私有包与依赖混淆

Dependency Confusion 常见于企业内部包名没有命名空间，攻击者在公网发布同名高版本包，构建系统误装公网包。

防护建议：

- 内部包使用组织命名空间，例如 `@company/ui`。
- `.npmrc` 明确 scope 对应私有 registry。
- CI 禁止从未授权 registry 拉取内部 scope 包。
- 内部包版本发布走统一流水线，避免手动发布。
- 关键依赖做来源校验和清单归档。

## 10. 第三方脚本治理

第三方脚本风险很高，因为它们直接运行在页面主线程，权限接近业务代码。

建议：

- 能 npm 构建进包的，不运行时从未知 CDN 拉取。
- 必须使用 CDN 时，固定版本并配置 SRI。
- 用 CSP 限制第三方脚本来源和外连目标。
- 对统计、客服、广告脚本做清单管理。
- 不让第三方脚本出现在支付、账号安全等高敏页面。
- 第三方脚本接入前评估其数据采集范围、外连域名、更新机制。

## 11. 发布前检查清单

| 检查项     | 要求                                |
| :--------- | :---------------------------------- |
| 前端产物   | 没有私钥、Token、密码、JWT secret   |
| Source Map | 不公开部署到公网                    |
| API 响应   | 不返回多余敏感字段                  |
| 日志上报   | Authorization、Cookie、PII 已脱敏   |
| lockfile   | 已提交且 CI 使用 frozen lockfile    |
| 依赖升级   | 经过 changelog、脚本、依赖树 review |
| 第三方 CDN | 固定版本并配置 SRI                  |
| CSP        | 限制脚本来源和接口外连目标          |
| 权限控制   | 服务端强制执行，前端只做体验优化    |

## 12. 实战建议

- 前端变量默认按公开信息处理，任何密钥都不要进入前端构建。
- 生产 Source Map 上传到监控平台后从公网产物中删除。
- 依赖升级进入代码 review，不把 lockfile 变化当作无意义噪音。
- 对第三方脚本建立资产清单，明确负责人、用途、域名、接入页面。
- 关键业务页面启用更严格的 CSP，不允许随意新增外部脚本。
- CI 增加敏感信息扫描、依赖审计、产物检查，尽量把风险挡在发布前。
