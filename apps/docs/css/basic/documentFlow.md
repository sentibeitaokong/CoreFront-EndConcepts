# 文档流

文档流是浏览器默认的排版引擎。理解文档流的物理规则，是解决所有 CSS 布局 Bug 的前提。当你觉得“CSS 不听话”时，通常是因为你不小心破坏了文档流，或者不理解它的隐形规则（如外边距合并、行高机制等）。

## 1. 布局基础：文档流与 Display 详解

理解文档流和 display 属性是掌握 CSS 布局的基石。

### 1.1 什么是“标准文档流” (Normal Document Flow)?

标准文档流（Normal Flow）是浏览器默认排列 HTML 元素的规则。如果你不写任何 CSS 布局代码（如 `float`, `position`, `flex`, `grid`），元素就会按照文档流进行排列。

**核心规则：**

- **从上到下**：块级元素（Block）会在垂直方向上一个接一个地排列。
- **从左到右**：行内元素（Inline）会在水平方向上从左向右排列，直到一行排满，自动换行到下一行。

### 1.2 Display 属性的核心模式

`display` 属性决定了元素在文档流中的行为方式（它是“**块**”还是“**行内**”？），以及它的子元素如何布局。

#### 1.2.1 `display: block` (块级元素)

块级元素是页面结构的骨架。

- **默认元素**：`<div>`, `<p>`, `<h1>`~`<h6>`, `<ul>`, `<li>`, `<section>`, `<article>` 等。
- **行为特征**：
  - **独占一行**：无论内容多少，它总是强制换行，左右两边不允许有其他元素（除非使用浮动或 Flex）。
  - **填满容器**：默认宽度（`width`）自动填满父容器的 100%。
  - **支持盒模型**：你可以自由设置 `width`, `height`, `margin`, `padding`，它们都会生效并占据空间。

#### 1.2.2 `display: inline` (行内元素)

行内元素主要用于包裹文本内容。

- **默认元素**：`<span>`, `<a>`, `<em>`, `<strong>`, `<label>` 等。
- **行为特征**：
  - **并排显示**：和其他行内元素在同一行显示，不会自动换行。
  - **宽度随内容**：宽度仅由其包含的内容决定，**无法**手动设置 `width` 和 `height`（设置了也无效）。
  - **盒模型受限**：
- **水平方向**：`margin-left`, `margin-right`, `padding-left`, `padding-right` **有效**，会推开其他元素。
- **垂直方向**：`margin-top`, `margin-bottom` **无效**。`padding-top`, `padding-bottom` 视觉上会有背景色延伸，但**不会推开**上下方的其他元素（不会影响文档流的高度）。

#### 1.2.3 `display: inline-block` (行内块元素)

这是结合了前两者优点的混合模式。在 Flexbox 流行之前，常用于制作导航栏或并排排列的卡片。

- **默认元素**：`<img>`, `<input>`, `<button>`, `<textarea>`。
- **行为特征**：
  - **对外像 Inline**：可以和其他元素并排在一行。
  - **对内像 Block**：可以完整支持盒模型，设置 `width`, `height`, `margin`, `padding` 均有效。

#### 1.2.4 `display: none` vs `visibility: hidden`

虽然它们都用于隐藏元素，但对文档流的影响完全不同。

| 属性                     | 描述         | 对文档流的影响                                                 | 重绘/回流                      |
| ------------------------ | ------------ | -------------------------------------------------------------- | ------------------------------ |
| **`display: none`**      | **完全移除** | 元素从文档流中消失，**不占据任何空间**。就像它从未存在过一样。 | 触发**回流 (Reflow)** 和重绘。 |
| **`visibility: hidden`** | **视觉隐藏** | 元素不可见，但在文档流中**依然占据原来的空间**（变成透明的）。 | 只触发**重绘 (Repaint)**。     |

## 2. BFC (块级格式化上下文){#bfc}

这是理解文档流最重要的高级概念。

**定义**：**BFC (Block Formatting Context)** 是页面上的一个独立的渲染区域。

**形象理解**：BFC 就像是一个**密封的箱子**。

- 箱子里的元素怎么乱跑（比如浮动），都不会影响到箱子外面的元素。
- 箱子外面的元素（比如 Margin），也穿透不进箱子内部。

### 2.1 如何触发 BFC？

只要给元素添加以下任意一个属性，就会创建 BFC：

- `overflow`: `hidden`、`auto` 或 `scroll` (最常用的方式)。
- `display`: `flex`、`grid`、`inline-block` 或 `flow-root` (现代标准推荐)。
- `position`: `absolute` 或 `fixed`。
- `float`: `left` 或 `right`。

### 2.2 BFC 的三大“超能力” (用途)

- **包含浮动元素** (解决高度塌陷)。
- **阻止外边距折叠** (解决 Margin Collapse)。
- **防止被浮动元素覆盖** (两栏自适应布局)。

## 3. IFC (行内格式化上下文){#ifc}

如果说 BFC 是管理“**块**”的密封箱子，那么 **IFC (Inline Formatting Context)** 就是管理“**文字和行内元素**”的**流水排版系统**。

**定义**：IFC 规定了行内级别盒子（如文字、`<span>`、`<img>`、`inline-block`）在容器中是如何进行水平和垂直排布的。

**形象理解**：IFC 就像老式打字机打字。字（元素）一个接一个从左向右紧密排列，碰到边界了，打字机“叮”一声换到下一行继续排。

### 3.1 IFC 的内部运作机制（行框与基线）

在 IFC 中，有几个看不见的“幽灵结构”在控制着布局：

1. **行框 (Line Box)**：
   - IFC 中的每一行文字/元素都会生成一个看不见的矩形框，叫作“行框”。
   - 行框的**宽度**由父容器决定（并受浮动元素挤压而变窄）。
   - 行框的**高度**由这一行里面**最高的那个元素**（或是文字的行高 `line-height`）决定。

2. **水平对齐 (`text-align`)**：
   - 当一行排不满时，行内盒子在行框里的水平位置由父容器的 `text-align` 属性控制（`left`, `center`, `right`, `justify`）。

3. **垂直对齐 (`vertical-align` 与 Baseline)**：
   - 在同一个行框里，高矮胖瘦不同的元素（比如大号字、小号字、图片）默认是按照**基线 (Baseline)** 对齐的，而不是底部对齐。
   - 你可以使用 `vertical-align` (`top`, `middle`, `bottom`, `baseline`) 来改变它们在行框内的上下位置。

## 4. 文档流中的常见“怪异”问题与底层原因

以下是开发者经常遇到的布局“灵异现象”，它们都源于 BFC 或 IFC 的底层渲染机制。

### 4.1 外边距合并 (Margin Collapse) —— [BFC 相关]

**现象**：

- **兄弟之间**：上下相邻的两个 div，分别有 `margin-bottom: 30px` 和 `margin-top: 20px`，最终间距只有 **30px** (取最大值)。
- **父子之间**：给子元素设置 `margin-top: 20px`，子元素没相对于父元素下移，而是**带着父元素一起掉下来了**。

**解决方案**：

| 场景                  | 解决方案                                                                                                                           | 备注                                                                                 |
| --------------------- | ---------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------ |
| **兄弟合并**          | 无法通过自身属性消除                                                                                                               | 通常不解决，接受这个设定，或者只给一个方向设 Margin。                                |
| **父子合并** (最常见) | 1. **给父元素触发 BFC** (如 `overflow: hidden` 或 `display: flow-root`) <br/>2. **给父元素加边框/Padding** (如 `padding-top: 1px`) | 只要父子之间有“**阻隔**”(边框/padding) 或父元素是“**密封箱子**”(BFC)，合并就会取消。 |

### 4.2 Inline-Block 元素之间的“幽灵空白” —— [IFC 相关]

**现象**：
当你把几个 `display: inline-block` 的元素横向排列时，即使 margin 设为 0，它们之间也总有 **4px 左右的间隙**。

**原因**：
因为它们参与了 **IFC 排版**。浏览器把 HTML 代码中的**换行符**（回车）当成了一个普通的“空格字符”渲染到了行框（Line Box）里。

**解决方案**：

- **方案 A (推荐)**：改用 **Flexbox** 布局（彻底脱离 IFC）。
- **方案 B (Hack)**：父元素 `font-size: 0`，消除空格字符的大小，子元素再恢复字体大小。

### 4.3 图片底部多出的“下巴” (3px 空隙) —— [IFC 相关]

**现象**：
在一个 div 里放一张 img，div 的高度会比 img 高出约 3-5px，导致底部有缝隙。

**原因**：
图片默认是 `inline` 元素，处于 **IFC 规则**中。IFC 默认按照**基线 (Baseline)** 对齐，而基线下方必须为英文字母（如 `g`, `y`, `p`）的“**尾巴**”留出空间。图片坐在了基线上，下方预留的空白就成了那个“**下巴**”。

**解决方案**：

- **方案 A (推荐)**：将图片脱离 IFC，设为块级。`img { display: block; }`
- **方案 B**：改变 IFC 中的对齐方式。`img { vertical-align: bottom; }`

### 4.4 高度塌陷 (Height Collapse) —— [BFC 相关]

**现象**：
父容器没有设置高度，里面的子元素全部 `float: left`。结果父容器**高度变成了 0**。

**原因**：
浮动元素脱离了标准文档流。父容器在计算高度时，忽略了浮动的子元素。

**解决方案**：

- **现代方案**：给父元素触发 BFC。`display: flow-root;`
- **经典 Clearfix**：使用伪元素闭合浮动。

### 4.5 单行文字的垂直居中黑魔法 —— [IFC 相关]

**现象**：
将元素的 `line-height` 设置得和 `height` 一样高，文字就会自动垂直居中。

**原因**：
这也是 **IFC** 的杰作。当你设置了 `line-height: 50px`，相当于强制把这一行的“**行框 (Line Box)**”撑到了 50px 高。在行框内部，文字默认是基于中轴线对称分布的，所以在视觉上就完美居中了。

## 5. 总结：如何避免 90% 的文档流问题？

- **拥抱现代布局**：能用 Flex/Grid 就别用 Float/Inline-Block。现代布局系统规避了几乎所有传统 BFC/IFC 的坑。
- **理解盒模型**：始终使用 `box-sizing: border-box`。
- **警惕 Margin 合并**：当父子元素边缘重合时，优先想到 BFC 或使用 Padding 代替 Margin。
- **图片初始化**：习惯性把布局用的图片设为 `display: block`，彻底告别 IFC 带来的神秘间隙。
