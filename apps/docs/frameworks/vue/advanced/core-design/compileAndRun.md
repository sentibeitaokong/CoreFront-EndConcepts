# 运行时与编译时

## 1. 基本概念

在程序设计与框架开发中，**编译时（Compile Time）** 与 **运行时（Run Time）** 是两个重要的阶段。

- **编译时**：源代码被转换为可执行代码（或中间表示）的阶段。在这个阶段，可以进行静态分析、优化、类型检查等操作。
- **运行时**：编译后的代码实际执行的阶段。在这个阶段，程序会响应输入、操作内存、与用户交互等。

对于前端框架而言，这两个阶段的分工决定了框架的性能特征、开发体验以及灵活性。

## 2. 框架中的两种典型策略

### 2.1 纯运行时框架

框架只在**运行时**工作，接收用户编写的配置（如 `render` 函数或 VNode 树），然后动态创建和更新 DOM。例如 React 早期的 `React.createElement` 和 Vue 的运行时版本（不带模板编译器）。

**特点**：

- 模板 / JSX 直接编译为 `render` 函数（编译时简单或不需要编译）。
- 所有优化依赖运行时 Diff 算法。
- 灵活性高，可动态生成任何 UI 结构。
- 运行时开销相对较大，因为需要在运行时解析和比较 VNode 树。

### 2.2 纯编译时框架

框架在**编译时**尽可能多地完成工作，生成直接操作 DOM 的命令式代码，运行时几乎不做 Diff。例如 Svelte。

**特点**：

- 模板在编译时转换为高效的 DOM 操作代码。
- 运行时体积极小（不需要虚拟 DOM 运行时）。
- 性能极佳，更新时直接修改 DOM。
- 灵活性受限：一些动态特性（如动态组件、递归模板）需要额外的运行时支持或无法实现。

### 2.3 编译时 + 运行时混合框架

框架在编译时对模板进行静态分析和优化，生成带有优化提示的 `render` 函数，运行时根据这些提示进行高效的更新。**Vue 3 正是这种混合策略的代表**。

## 3. Vue 的编译时与运行时分工

### 3.1 编译时（Compiler）

Vue 的模板编译器（`@vue/compiler-dom` 或 `@vue/compiler-core`）在构建阶段（或运行时动态编译）执行以下任务：

| 优化技术         | 说明                                                                         |
| ---------------- | ---------------------------------------------------------------------------- |
| **模板解析**     | 将模板字符串解析为 AST（抽象语法树）。                                       |
| **静态提升**     | 将不变的节点（纯文本、静态元素）提升到渲染函数外部，避免重复创建 VNode。     |
| **PatchFlags**   | 为动态节点标记更新类型（如 TEXT、CLASS、STYLE、PROPS），运行时只需检查标志。 |
| **Block 树**     | 识别动态节点，生成 `openBlock` / `createBlock` 调用，收集动态子节点。        |
| **缓存内联函数** | 缓存内联的事件处理函数，避免子组件不必要的重渲染。                           |
| **Tree-shaking** | 仅引入运行时需要的 API（如 `createVNode`、`openBlock`）。                    |

**编译输出示例**：

```javascript
// 模板: <div>{{ msg }}</div>
// 编译后（简化）:
import {
  toDisplayString as _toDisplayString,
  createVNode as _createVNode,
  openBlock as _openBlock,
  createBlock as _createBlock,
} from 'vue'

const _hoisted_1 = /*#__PURE__*/ _createVNode(
  'div',
  null,
  'static text',
  -1 /* HOISTED */,
)

export function render(_ctx, _cache) {
  return (
    _openBlock(),
    _createBlock('div', null, [
      _hoisted_1,
      _createVNode('p', null, _toDisplayString(_ctx.msg), 1 /* TEXT */),
    ])
  )
}
```

### 3.2 运行时（Runtime）

Vue 运行时（`@vue/runtime-core` 和 `@vue/runtime-dom`）负责：

- **响应式系统**：通过 `Proxy` 追踪依赖，在数据变化时触发更新。
- **虚拟 DOM**：创建、挂载、更新、销毁 VNode 树。
- **Diff 算法**：在更新时比较新旧 VNode，最小化 DOM 操作。
- **调度器**：将多个更新合并为一次异步任务，优化性能。
- **组件系统**：管理组件实例、生命周期、插槽等。

运行时利用编译时生成的优化提示（如 `PatchFlags`、静态提升节点、Block 树）来跳过不必要的 Diff 工作，实现靶向更新。

## 4. 编译时 vs 运行时：权衡对比

| 维度           | 编译时侧重（如 Svelte）      | 运行时侧重（如 React）     | 混合策略（Vue 3）                |
| -------------- | ---------------------------- | -------------------------- | -------------------------------- |
| **性能**       | 极高（直接 DOM 操作）        | 中等（需要 Diff）          | 高（编译优化 + 智能 Diff）       |
| **运行时体积** | 极小（无需虚拟 DOM 库）      | 较大（包含 Diff 和调度器） | 中等（可 tree-shaking）          |
| **灵活性**     | 较低（动态特性受限）         | 极高（完全动态）           | 高（保持动态能力，同时优化静态） |
| **编译复杂度** | 高（编译器需要生成精确代码） | 低（JSX 简单转换）         | 中（需要实现多种优化策略）       |
| **开发体验**   | 良好（模板简洁）             | 优秀（JSX 灵活）           | 优秀（模板或 JSX 都可选）        |

## 5. Vue 选择混合策略的原因

1. **兼顾性能与灵活性**：编译时优化消除了大部分运行时开销，同时保留了动态能力（如动态组件、插槽、递归组件）。
2. **渐进式增强**：用户可以选择仅使用运行时版本（如果已预编译模板），或者在需要时动态编译模板。
3. **开发体验友好**：开发者编写标准模板或 JSX，无需关心底层优化细节。
4. **体积可控**：通过 tree-shaking，未使用的运行时特性不会被打包。
5. **生态兼容**：既支持构建时编译（Vite / Webpack 插件），也支持运行时编译（独立构建版本）。

## 6. 总结

- **编译时**负责静态分析和提前优化，生成带有提示信息的渲染代码。
- **运行时**负责响应式依赖追踪和高效的 DOM 更新，利用编译时的提示避免不必要的工作。
- Vue 3 的混合策略使其在性能、体积、灵活性、开发体验之间达到了优秀的平衡，成为现代前端框架的主流设计之一。
